# Stage 3: Act (`/dev:pair act`)

**Purpose**: Execute implementation plan with validation, maintaining traceability to specification.

## Usage

```
/dev:pair act [PLAN_FILE] [--team-mode] [EXTRA_INSTRUCTIONS]
```

## Workflow

1. **Read Plan & Detect Resume**
   - Load implementation plan from `PLAN_FILE`
   - Check current `Stage:` value in the spec file
   - **If `Stage: Act`** (resume): scan for checkpoint (see [Checkpoint + Resume](#checkpoint--resume) below)
   - **If not `Act`** (fresh start): update `Stage:` to `Act` and `Last Updated:` to today's date
   - Detect change type: Bug Fix | Greenfield | Brownfield

2. **Execute Phases Sequentially**
   - If resuming, skip to the resume point (first `- [ ]` step)
   - Each phase must complete without breaking system
   - Follow step-by-step file actions exactly
   - After completing all steps in a phase, run the phase's **Verification** section (tests, linter, build, etc.)
   - Do NOT proceed to the next phase until verification passes
   - If verification fails, fix the issue within the current phase before moving on

3. **Track Progress in Plan File (update as you go)**
   - Update `- [ ]` to `- [x]` after each completed step
   - Update phase status when all steps done
   - If implementation deviated from a step, add a brief inline note explaining what changed
   - If a step was skipped, mark it `- [~]` with a reason
   - If an unplanned step was needed, insert it in the plan with `[UNPLANNED]` prefix and check it off
   - **Write these updates to the plan file immediately after each step**, not at the end — the plan file should always reflect current progress

4. **Type-Specific Approach**

   **Bug Fix:**
   - START with regression test that reproduces bug
   - VERIFY test fails before fix
   - IMPLEMENT minimal fix
   - VERIFY test passes after fix
   - CHECK no side effects

   **Greenfield:**
   - FOLLOW architecture from plan
   - ESTABLISH test coverage from start (>80%)
   - CREATE documentation as you build
   - VALIDATE against design patterns

   **Brownfield:**
   - VERIFY existing behavior before changes
   - IMPLEMENT behind feature flags when possible
   - MAINTAIN backward compatibility
   - TEST both old and new code paths

5. **Validate Against Acceptance Criteria**
   - Code meets all success criteria from spec
   - Tests validate spec requirements
   - No architectural drift from plan

6. **Validate Success Criteria & Milestones**
   - Review all success criteria defined in the spec
   - Verify each milestone has been achieved
   - For each criterion/milestone, confirm:
     - Implementation satisfies the requirement
     - Tests prove the requirement is met
     - No regression in existing functionality
   - If any criterion is not met, either:
     - Complete the missing work before finalizing
     - Document why criterion was descoped (requires user approval)

7. **Finalize Stage**
   - When all phases are completed, verified, and success criteria validated
   - Update `Stage:` to `Done` and `Last Updated:` to today's date in the spec file
   - If in `--team-mode`, delegate the status update to a teammate

## Checkpoint + Resume

When `/dev:pair act` is invoked on a spec where `Stage:` is already `Act`, treat it as a **resume** rather than a fresh start.

### Detection

1. Read `Stage:` from the spec file header
2. If `Stage: Act` → this is a resume
3. If anything else (`Plan`, `Init`, etc.) → this is a fresh start

### Resume Algorithm

```
1. Scan all steps in the plan for their checkbox status
2. Find the first unchecked step: `- [ ]`
3. Determine which phase it belongs to
4. Report resume point to user:
   "Resuming from Phase N, Step N.M: <step_name>
    Previously completed: X/Y steps across Z phases"
5. Skip to that step and continue execution normally
```

### Rules

- **`- [x]` (completed):** Skip entirely — do NOT re-execute
- **`- [~]` (skipped):** Skip entirely — already marked with reason
- **`- [ ]` (unchecked):** First one found is the resume point
- **`[UNPLANNED]` steps:** Treat like any other step based on checkbox state
- **Phase verification:** If resuming mid-phase, do NOT re-run verification for prior phases. Only run verification for the current phase once all its steps complete.
- **Stage header:** Do NOT overwrite `Stage: Act` on resume — only update `Last Updated:`

### Edge Cases

- **All steps checked:** Proceed directly to step 5 (Validate Against Acceptance Criteria)
- **No steps checked but Stage is Act:** The previous run failed before completing any step — start from Phase 1, Step 1
- **Verification failed on prior run:** The last checked step might have introduced a broken state. Run the current phase's verification before continuing to confirm the system is healthy.
