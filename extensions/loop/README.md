# Cycle Runner — Persistent Agent Loop Extension

A pi coding agent extension that adds the `/loop` command, keeping the agent working across multiple turns until a specified exit condition is satisfied.

## What It Does

Once activated, the Cycle Runner re-injects a follow-up prompt after every completed agent turn, driving continuous progress. The cycle ends when the agent calls the `complete_loop` tool — signalling that the exit condition has been fully met.

Typical use cases:

- **Test-driven iteration** — run, fix, repeat until the full test suite is green
- **Conditional workflows** — express any natural-language condition and let the agent work toward it
- **Open-ended tasks** — hand control to the agent and let it decide when the work is genuinely done

## Quick Start

```
/loop tests                          # Loop until all tests pass
/loop self                           # Agent decides when it is finished
/loop custom no TypeScript errors    # Loop until a specific condition is met
```

Run `/loop` with no arguments (TUI mode only) to open an interactive strategy picker.

## Strategies

| Strategy | Command                    | Behaviour                                                   |
| -------- | -------------------------- | ----------------------------------------------------------- |
| `tests`  | `/loop tests`              | Runs the test suite every turn; exits when all tests pass   |
| `custom` | `/loop custom <condition>` | Loops until the stated condition is satisfied               |
| `self`   | `/loop self`               | The agent autonomously determines when the task is complete |

## Interactive Mode

In TUI sessions, running `/loop` without arguments opens a bordered strategy picker:

```
/loop
```

Arrow keys navigate the list; **Enter** confirms. Selecting `custom` opens an inline editor for the exit condition text. Press **Esc** at any point to cancel without starting a cycle.

If a cycle is already running you will be asked to confirm before it is replaced.

## Status Widget

While a cycle is active a status line appears in the header showing the current iteration count and a short description of the exit condition:

```
Cycle running: tests pass (iteration 4)
```

The label is generated by a small LLM call (Haiku preferred). If that call fails, a static fallback is used instead.

## Stopping a Cycle

A cycle stops when any of the following happens:

1. **The agent calls `complete_loop`** — the normal completion path
2. **A turn is aborted** (Ctrl-C) — you are prompted to confirm stopping the cycle
3. **A new cycle is started** — replaces the existing one after confirmation

## How It Works

### Turn lifecycle

After each agent turn (`agent_end`), the extension checks:

- Is a cycle active? If not, skip.
- Was the last assistant message aborted? If so, prompt the user.
- Otherwise: increment the iteration counter, persist updated state, refresh the widget, and send the next follow-up prompt.

### State persistence

Cycle state is written as a custom session entry so it survives reloads and session switches:

```ts
type CycleState = {
  active: boolean;
  mode?: "tests" | "custom" | "self";
  condition?: string; // exit condition text (custom mode only)
  prompt?: string; // follow-up message injected each iteration
  summary?: string; // LLM-generated short label for the widget
  loopCount?: number; // increments with each iteration
};
```

The extension always reads the **most recent** matching entry, so compactions and reloads are handled correctly.

### Context compaction

When pi compacts conversation history, the extension hooks `session_before_compact` and injects an extra instruction to keep cycle context alive in the summary:

```
Cycle active. Exit condition: <condition>. Retain this cycle state and exit condition in the compacted summary.
```

This prevents the agent from losing awareness of the running cycle after a compaction.

### Session restore

Both `session_start` and `session_switch` trigger state rehydration, which:

1. Reads the latest cycle state from the session entry log
2. Restores the status widget
3. Re-generates the condition label asynchronously if it was never saved

## Events Used

| Event                    | Purpose                                             |
| ------------------------ | --------------------------------------------------- |
| `agent_end`              | Schedule the next iteration or ask the user to stop |
| `session_before_compact` | Inject cycle context into the compaction prompt     |
| `session_start`          | Rehydrate cycle state on pi startup                 |
| `session_switch`         | Rehydrate cycle state when switching sessions       |

## Headless / No-TUI Mode

When pi is running piped or headless, the interactive selector is unavailable. Pass the strategy directly:

```
/loop tests
/loop self
/loop custom all API endpoints return 200
```

Invoking `/loop` with no arguments in headless mode prints a usage hint and returns.

## Notes

- The `complete_loop` tool takes no parameters — the agent simply calls it when ready.
- The extension is abort-safe: interrupted turns prompt the user rather than continuing silently.
- Summary generation is non-blocking; the widget falls back to a static label until the LLM responds.
- When running without a UI, confirmation prompts default to `true` (proceed).
